# 步骤 0: 加载 GPT-2 模型与分词器 —— 问题记录

---

## Q1: 词汇表是哪来的？是每个模型自带词汇表吗？

**A:** 词汇表**不是模型自带**的，而是由训练前的**分词器（Tokenizer）**决定的。

- **字符级**（本演示用的简化版）：词汇表 = 所有可能出现的字符，如 a-z + 空格
- **BPE**（真实 GPT-2 用的）：在训练语料上统计高频子词对，反复合并得到词汇表
  - GPT-2 的词汇表有 **50257** 个 token，包含常见单词片段如 `"ing"`, `"the"`, `"Ġworld"`

**分词器的训练过程（以 BPE 为例）：**
1. 初始词汇表 = 所有单个字符
2. 统计语料中相邻 token 对的出现频率
3. 合并出现频率最高的一对，加入词汇表
4. 重复 2-3 步直到词汇表达到目标大小
5. 最终词汇表保存为文件（如 `vocab.json`），模型训练时加载使用

**结论**：词汇表是**训练前由分词器算法从语料中学出来的**，模型只负责在词汇表范围内预测下一个 token 的索引。

## Q2: `transformers` 这个包是谁提供的？为什么会有 `GPT2LMHeadModel` 这样特殊的名字？

**A:** `transformers` 库由 **Hugging Face**（https://huggingface.co）提供。

它虽然名字叫 "transformers"，但定位是**预训练模型的大集合**，而非纯算法工具包：
- Hugging Face 是 AI 界的 GitHub，托管超过 100 万个模型
- `transformers` 库是配套 SDK，用统一 API 让你加载这些模型

**类名拆解：`GPT2LMHeadModel`**

| 部分 | 含义 |
|------|------|
| `GPT2` | 模型架构（基于 GPT-2 论文） |
| `LMHead` | Language Modeling Head —— 在隐藏状态上加输出层，预测下一个 token |
| `Model` | 基础 PyTorch `nn.Module` |

同一个 GPT-2 架构，Hugging Face 还提供了其他变体：
- `GPT2Model`：只有 Transformer 主体，无输出头（用于特征提取）
- `GPT2LMHeadModel`：+ LM Head，用于文本生成
- `GPT2ForSequenceClassification`：+ 分类头，用于情感分析
- `GPT2ForTokenClassification`：+ token 级分类头，用于 NER

## Q3: `LMHead` 这个 Head 是什么意思？为什么叫"头"、"输出头"？

**A:** **Head = 输出头 = 接在模型主干后面的"任务专用输出层"**

```
输入文本 ──→ [ Backbone 主干网络 ] ──→ [ Head 输出头 ] ──→ 最终结果
            (Transformer 12层)          (一个线性层)        (下一个词的概率)
                  ↑                        ↑
            提取通用特征              把特征转成任务需要的格式
```

**为什么叫"头"？**

| 比喻 | 解释 |
|------|------|
| **动物身体** | Backbone（脊柱/主干）是身体，Head（头）负责"看、说、做"不同的事 |
| **水龙头** | 同一根水管（Backbone），接不同的头：淋浴头、水龙头、花洒 |
| **USB 转接头** | 电脑是 Backbone，接 USB-A 头、USB-C 头、HDMI 头做不同的事 |

**具体到 GPT-2：**

```python
GPT2Model                    # 只有主干（12层 Transformer）
    └── 输出 shape: (seq_len, 768) 隐藏状态向量
    
GPT2LMHeadModel = GPT2Model + LMHead    # 主干 + 语言建模头
    └── LMHead 就是一个线性层: 768 → 50257（词汇表大小）
    └── 输出 shape: (seq_len, 50257) 每个词的预测概率
```

**代码层面的真相**："Head" 在代码里通常就是**一个线性层**（`nn.Linear`）：

```python
# GPT2LMHeadModel 的核心
self.lm_head = nn.Linear(n_embd, vocab_size, bias=False)  # 768 -> 50257
# GPT-2 更省参数：直接复用 wte（word embedding）的转置作为这个线性层
```

**结论**：Head 不神秘，它就是 Backbone 和最终输出之间的**一层"翻译器"**——把高维特征翻译成任务需要的格式（词汇表概率、类别概率、实体标签...）。

## Q4: transformers 库里除了 GPT 还有哪些模型？这些模型都是 Hugging Face 首创的吗？它是定期把业界流行的模型吸收进来吗？

**A:** transformers 库支持 **1000+ 种不同的模型架构**，远不止 GPT：

| 来源机构 | 代表模型 |
|---------|---------|
| **OpenAI** | GPT, GPT-2 |
| **Google** | BERT, T5, BART, PaLM |
| **Meta** | LLaMA, LLaMA-2, LLaMA-3, OPT |
| **Mistral AI** | Mistral, Mixtral (MoE) |
| **阿里巴巴** | Qwen, Qwen-2, Qwen-2.5 |
| **清华/智谱** | ChatGLM, ChatGLM-2, ChatGLM-3, GLM-4 |
| **百度** | Ernie, Ernie-Bot |
| **Stability AI** | Stable Diffusion (图像生成) |
| **OpenAI** | Whisper (语音识别), CLIP (图文理解) |
| **Meta** | SAM (图像分割), DINOv2 |

**这些模型都是 Hugging Face 首创的吗？**

**绝大部分不是。** Hugging Face 的定位是**"AI 模型基础设施平台"**，而不是模型研发实验室。

```
Google 发明 BERT ──→ 发布论文 + 官方代码/权重
       │
       ▼
Hugging Face 拿到官方权重 ──→ 转换成 safetensors 格式
       │
       ▼
Hugging Face 写适配代码 ──→ 加入 transformers 库
       │
       ▼
你调用: from transformers import BertModel   (统一 API)
```

**各角色的分工：**

| 角色 | 做什么 |
|------|--------|
| **研究机构/公司**（OpenAI, Google, Meta, 阿里...） | 设计架构、训练模型、发布论文和官方权重 |
| **Hugging Face** | 做"中间商"：统一格式、写适配层、提供下载和推理工具 |
| **开源社区** | 提交 PR、补充新模型支持、修复 bug |

**Hugging Face 吸收新模型的三种方式：**

1. **官方主动跟进**：核心热门模型（如 LLaMA）发布时，官方团队第一时间写适配代码
2. **社区贡献**：模型作者或开发者提交 PR 到 transformers 库
3. **Hub 直接托管**：不需要进库也能用，`AutoModel` 可根据配置文件自动识别架构

## Q5: Hugging Face Hub 上的模型是纯模型数据，还是包括代码？运行模型的代码是否已经在 transformers 库里包全了？动态下载的模型也可能包含如何运行自身的代码吗？

**A:** Hub 上主要是**纯数据（权重 + 配置）**，不是可执行程序。

### Hub 上的模型文件结构

```
模型目录/
├── config.json              ← 架构配置（层数、维度、头数等）
├── pytorch_model.bin        ← 或 model.safetensors，纯权重数据
├── tokenizer.json           ← 分词器词汇表
├── tokenizer_config.json    ← 分词器配置
└── (可选) modeling_xxx.py   ← 自定义模型代码（部分模型才有）
```

### 运行代码从哪来？分两种情况

**情况一：标准架构（约 95% 的模型）**

代码在 transformers 库里，权重从 Hub 下载：

```python
from transformers import GPT2LMHeadModel

# 1. 从 transformers 库里找 GPT2LMHeadModel 的类定义（本地就有）
# 2. 从 Hugging Face Hub 下载 gpt2 的权重文件
# 3. 用本地代码 + 远程权重实例化模型
model = GPT2LMHeadModel.from_pretrained("gpt2")
```

| 来源 | 内容 |
|------|------|
| **transformers 库（本地安装）** | `GPT2LMHeadModel` 类的 PyTorch 代码 |
| **Hugging Face Hub（远程下载）** | `pytorch_model.bin` 纯权重数据 |

**情况二：自定义架构（约 5%，越来越常见）**

代码不在 transformers 库里，而是**随模型一起从 Hub 下载**：

```python
from transformers import AutoModel

# trust_remote_code=True 允许执行从 Hub 下载的 .py 文件
model = AutoModel.from_pretrained(
    "某个社区新架构",
    trust_remote_code=True   # ← 关键！允许运行远程代码
)
```

此时 Hub 上的模型目录会包含自定义代码：

```
模型目录/
├── config.json
├── pytorch_model.bin
├── modeling_my_custom_arch.py   ← 自定义模型代码
└── ...
```

`config.json` 中会有自动映射配置：
```json
{
  "auto_map": {
    "AutoModel": "modeling_my_custom_arch.MyCustomModel"
  }
}
```

transformers 读取到这段配置后，会自动去 Hub 下载 `modeling_my_custom_arch.py` 并动态加载其中的类。

### 两种方式的对比

| 维度 | 标准架构（如 GPT-2） | 自定义架构（如早期 ChatGLM） |
|------|---------------------|---------------------------|
| **模型代码在哪** | transformers 库里 | Hub 随权重一起下载 |
| **需要联网吗** | 只需下载权重 | 需要下载权重 + 代码 |
| **是否安全** | 安全（代码是审过的） | 需谨慎（`trust_remote_code=True` 等于允许运行陌生代码） |
| **更新方式** | 升级 transformers 包 | 模型作者更新 Hub 文件即可 |

### 类比理解

| 类比 | 解释 |
|------|------|
| **标准架构** | 买游戏主机（transformers 库）+ 买游戏卡带（Hub 权重）。主机自带运行代码，卡带只存数据。 |
| **自定义架构** | 买 PC（transformers 库）+ 下载 Steam 游戏（Hub 完整包）。游戏本身包含可执行程序，PC 只提供运行环境。 |

## Q6: 模型的层数和块数是一回事吗？

**A:** 在 GPT-2 的语境下，**是一回事**。

| 术语 | 含义 | GPT-2 的值 |
|------|------|-----------|
| **层数 (n_layer)** | Transformer Block 的数量 | 12 |
| **块数 (n_block)** | 同上，只是叫法不同 | 12 |

```python
for i in range(N_LAYER):      # N_LAYER = 12
    block = transformer.h[i]  # 第 i 个 block
    # 每个 block = Attention + FFN + 2个 LayerNorm + 2个残差连接
```

但要注意，如果把"层"理解得更细，每个 Block 内部还有多个**子层**：
- LayerNorm（2个）
- Self-Attention（1个）
- Feed-Forward（1个）

所以：
- 按 **Block** 算：12 层
- 按 **子层** 算：约 48 个操作

**不同来源的叫法**：GPT-2 论文说"12 layers"，PyTorch 代码里叫 `self.h`（blocks 列表），Hugging Face 配置叫 `n_layer`，意思都一样。

## Q7: `transformer.wte` 里的 `wte` 是什么意思？

**A:** 是缩写：

| 缩写 | 全称 | 作用 |
|------|------|------|
| **`wte`** | **W**ord **T**oken **E**mbedding | 词嵌入矩阵，把 token ID 映射成向量 |
| **`wpe`** | **W**ord **P**osition **E**mbedding | 位置嵌入矩阵，给每个位置加上位置信息 |

```python
self.wte = nn.Embedding(vocab_size, n_embd)   # (50257, 768)
self.wpe = nn.Embedding(n_positions, n_embd)  # (1024, 768)
```

`c_` 前缀（如 `c_attn`, `c_proj`）代表 **conv1d**，GPT-2 用 1D 卷积实现线性投影，效果上等价于 `Linear`。

## Q8: `wpe` 里除了 weight 还有什么信息？

**A:** `wpe` 是 PyTorch `nn.Embedding` 对象，除了 `weight` 还有一些配置参数，但大多未启用：

```python
wpe.num_embeddings      # 1024  ← 最大支持的位置数 ✅
wpe.embedding_dim       # 768   ← 向量维度 ✅
wpe.weight              # (1024, 768)  ← 实际参数 ✅

wpe.max_norm            # None  ← 未启用
wpe.norm_type           # 2.0   ← 默认值，未生效
wpe.padding_idx         # None  ← 未启用
wpe.scale_grad_by_freq  # False ← 未启用
wpe.sparse              # False ← 未启用
```

**唯一有价值的信息是 `num_embeddings=1024`**——它告诉你模型最多能处理多长的序列。

## Q9: "GPT-2 使用权重共享：输出投影矩阵 = wte 的转置"是什么意思？

**A:** GPT-2 把**输入嵌入矩阵**和**输出投影矩阵**复用为同一个权重。

**正常情况（不共享）**：
- `wte`: (50257, 768) → 输入用，约 3850万 参数
- `W_out`: (768, 50257) → 输出用，约 3850万 参数
- **合计约 7700万 参数**

**GPT-2 的做法**：
- 只用 `wte` 一个矩阵
- 输入时：用 `wte` 正向查表（token ID → 向量）
- 输出时：用 `wte.T` 做投影（向量 → 词汇表 logits）
- **只需 3850万 参数，节省 50%**

```python
self.lm_head = nn.Linear(n_embd, vocab_size, bias=False)
self.lm_head.weight = self.wte.weight   # ← 共享同一个内存对象
```

**为什么有效**：输入和输出在同一个语义空间里操作，共享权重让语义理解保持一致，训练更稳定。

## Q10: `wte` 和 `wpe` 与模型是绑定的吧？不同模型的 block 参数，都是和 wte/wpe 配套训练出来的？换 wte 会怎样？

**A:** **完全正确，所有参数是一起"配套"训练出来的。**

GPT-2 的 1.24亿 参数在同一次反向传播中联合更新：
- `wte` 定义了语义空间的坐标系
- `wpe` 定义了位置编码方式
- 12 个 Block 基于这个坐标系建立了所有概念之间的关系

**换 wte 会怎样？** ❌ **完全错乱**

```python
# 把 GPT-2 的 wte 换成 BERT 的
输入 "hello" → BERT 的 wte → GPT-2 的 Block → 输出 = 垃圾
```

因为后面的 Block 期望的是 GPT-2 风格的语义空间，而输入给的是 BERT 风格的，互相"听不懂"。

**类比**：wte 是方言词典，Block 是用这个方言思考的大脑。换了粤语词典给普通话大脑，无法沟通。

**什么情况下可以换 wte？**
- ✅ 同一模型的微调版本（所有参数一起重新训练）
- ✅ 扩展词汇表（新增的行随机初始化后，整体继续训练）
- ❌ 跨模型混搭（如 GPT-2 + BERT）→ 完全不可行
